# 비기능적 요구사항

## 1. 성능
1. 파일 업로드 URL 발급 요청은 빠르게 응답해야 한다.
2. 대용량 파일 업로드는 서버를 경유하지 않고 파일 저장소로 직접 업로드할 수 있어야 한다.
3. 폴더 목록 조회는 대량 데이터에서도 응답 지연을 최소화해야 한다.

## 2. 확장성
1. 파일 저장소는 로컬 파일 서버, S3 등으로 교체 가능해야 한다.
2. 파일 업로드 완료 후 후처리는 비동기 이벤트 기반으로 확장 가능해야 한다.
3. 썸네일 생성, 바이러스 검사, 메타데이터 추출 같은 후처리를 독립적으로 추가할 수 있어야 한다.

## 3. 안정성
1. 업로드 완료 처리 중 장애가 발생해도 파일 메타데이터의 상태가 불일치하지 않도록 해야 한다.
2. 비동기 처리 실패 시 재시도 또는 실패 상태 관리가 가능해야 한다.
3. 파일 삭제, 복구, 영구 삭제는 중복 요청에도 안전하게 처리되어야 한다.

## 4. 보안
1. 사용자는 권한이 있는 파일 및 폴더에만 접근할 수 있어야 한다.
2. 업로드 URL과 다운로드 URL은 제한된 시간 동안만 유효해야 한다.
3. 파일 접근, 다운로드, 삭제 요청은 인증된 사용자 기준으로 검증되어야 한다.

## 5. 일관성
1. 파일 메타데이터와 실제 파일 저장소의 상태는 최종적으로 일치해야 한다.
2. 업로드 완료 이벤트를 기준으로 파일 상태가 갱신되어야 한다.
3. 삭제된 파일은 일반 목록 조회에서 제외되어야 한다.

## 6. 유지보수성
1. 도메인 규칙은 도메인 계층에 위치해야 한다.
2. 파일 저장소, 권한 검증, 이벤트 발행 등 외부 기술 의존성은 인프라 계층으로 분리해야 한다.
3. Gradle build-logic을 통해 공통 빌드 설정을 재사용할 수 있어야 한다.

## 7. 테스트 용이성
1. 핵심 도메인 로직은 외부 인프라 없이 단위 테스트할 수 있어야 한다.
2. JPA Repository는 실제 DB와 유사한 환경에서 통합 테스트할 수 있어야 한다.
3. Testcontainers를 활용하여 DB 의존 테스트의 신뢰성을 높여야 한다.

## 8. 관측 가능성
1. 주요 요청에는 추적 가능한 로그를 남겨야 한다.
2. 업로드, 다운로드, 삭제, 권한 실패 같은 주요 이벤트는 감사 로그로 추적 가능해야 한다.
3. 장애 발생 시 원인을 파악할 수 있도록 에러 코드와 traceId를 제공해야 한다.
